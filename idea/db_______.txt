[Pharmacy Backend API 명세]

Base URL:
http://172.23.0.128:8000/api/

주의:
- IP는 백엔드 PC에서 hostname -I로 확인한 현재 IP를 사용해야 함.
- 백엔드 PC에서만 테스트할 때는 http://127.0.0.1:8000/api/ 사용 가능.
- 현재 min_stock은 삭제됨.
- 리필 필요 여부는 처방전 등록 시 각 약의 quantity와 medicine.dispensing_stock을 비교해서 판단함.


1. 약 목록 조회

GET /api/medicine/

용도:
- 약 선택 드롭다운
- 재고 현황
- 약 정보 조회

응답 예시:
[
  {
    "id": 1,
    "medicine_name": "타이레놀",
    "storage_location": "123,456",
    "dispensing_location": "234,567",
    "lid_type": "hole",
    "storage_stock": 300,
    "dispensing_stock": 100
  }
]

필드 의미:
- medicine_name: 약 이름
- storage_location: 적재소 위치
- dispensing_location: 조제기 위치
- lid_type: 뚜껑 타입
- storage_stock: 적재소 재고
- dispensing_stock: 조제기 내부 재고


2. 처방전 등록

POST /api/prescriptions/

용도:
- 프론트에서 처방전을 등록할 때 사용

요청 Body 예시:
{
  "prescription_name": "김철수",
  "items": [
    {
      "medicine_name": "타이레놀",
      "quantity": 5
    },
    {
      "medicine_name": "탁센",
      "quantity": 1
    }
  ]
}

처리 기준:
- 각 약마다 quantity와 dispensing_stock을 비교함.
- quantity > dispensing_stock 이면 해당 EventItem.status = REFILL_REQUIRED
- quantity <= dispensing_stock 이면 해당 EventItem.status = READY
- 하나라도 REFILL_REQUIRED가 있으면 Event.status = REFILL_REQUIRED
- 전부 READY이면 Event.status = WAITING

성공 응답 예시: 전부 제조 가능
{
  "event_id": 1,
  "prescription_name": "김철수",
  "status": "WAITING",
  "refill_required": false,
  "refill_required_items": [],
  "message": "처방전이 등록되었고 제조 대기 상태입니다."
}

성공 응답 예시: 일부 약 리필 필요
{
  "event_id": 2,
  "prescription_name": "이영희",
  "status": "REFILL_REQUIRED",
  "refill_required": true,
  "refill_required_items": [
    {
      "medicine_name": "타이레놀",
      "requested_quantity": 5,
      "current_dispensing_stock": 3,
      "shortage": 2
    }
  ],
  "message": "처방전이 등록되었지만 일부 약 항목에 리필이 필요합니다."
}


3. 처방전 / 제조 현황 조회

GET /api/events/

용도:
- 제조 현황 화면
- 대시보드
- 처방전 전체 상태 확인
- 처방전 안의 약별 리필 필요 여부 확인

응답 예시:
[
  {
    "id": 2,
    "prescription_name": "이영희",
    "status": "REFILL_REQUIRED",
    "created_at": "2026-07-07T12:30:00Z",
    "items": [
      {
        "id": 1,
        "medicine_name": "타이레놀",
        "quantity": 5,
        "order": 1,
        "status": "REFILL_REQUIRED"
      },
      {
        "id": 2,
        "medicine_name": "탁센",
        "quantity": 1,
        "order": 2,
        "status": "READY"
      }
    ]
  }
]

Event.status 의미:
- REFILL_REQUIRED: 처방전 안에 리필 필요한 약이 있음
- WAITING: 제조 가능, 제조 대기
- PROCESSING: 제조중
- DONE: 제조 완료

EventItem.status 의미:
- READY: 이 약은 조제기 재고로 처리 가능
- REFILL_REQUIRED: 이 약은 조제기 재고 부족으로 리필 필요


4. 다음 제조 작업 조회

GET /api/tasks/next/

용도:
- ROS2 bridge_node가 다음 제조 작업을 가져갈 때 사용
- 프론트보다는 ROS2용 API

중요:
- WAITING 상태인 Event만 가져옴.
- REFILL_REQUIRED 상태인 Event는 아직 제조 불가능하므로 가져오지 않음.

응답 예시:
{
  "event_id": 1,
  "prescription_name": "김철수",
  "status": "WAITING",
  "items": [
    {
      "medicine_name": "타이레놀",
      "quantity": 2,
      "order": 1,
      "item_status": "READY",
      "refill_needed": false,
      "storage_location": "123,456",
      "dispensing_location": "234,567",
      "lid_type": "hole",
      "storage_stock": 300,
      "dispensing_stock": 98
    }
  ]
}

대기 작업이 없을 때:
HTTP 404
{
  "message": "대기 중인 작업이 없습니다."
}


5. 작업 상태 변경

POST /api/tasks/status/

용도:
- Event 전체 상태를 변경할 때 사용
- 주로 ROS2 bridge_node가 사용
- 프론트에서 수동 변경 버튼을 만들 경우에도 사용 가능

요청 Body 예시: 제조중으로 변경
{
  "event_id": 1,
  "status": "PROCESSING"
}

요청 Body 예시: 제조 완료로 변경
{
  "event_id": 1,
  "status": "DONE"
}

가능한 status:
- REFILL_REQUIRED
- WAITING
- PROCESSING
- DONE

성공 응답 예시:
{
  "message": "상태 변경 완료",
  "event_id": 1,
  "status": "PROCESSING"
}


6. 리필 완료 처리

POST /api/tasks/refill/

용도:
- 로봇이 적재소에서 약을 가져와 조제기에 리필 완료했을 때 사용
- 주로 ROS2 bridge_node가 사용
- 수동 리필 완료 버튼을 만들 경우 프론트도 사용 가능

요청 Body 예시:
{
  "medicine_name": "타이레놀",
  "amount": 20
}

내부 처리:
- storage_stock -= amount
- dispensing_stock += amount
- REFILL_REQUIRED 상태였던 Event들을 다시 검사함.
- EventItem.quantity <= Medicine.dispensing_stock 이면 EventItem.status = READY
- Event 안의 모든 EventItem.status가 READY가 되면 Event.status = WAITING으로 변경됨.

성공 응답 예시:
{
  "message": "리필 완료",
  "medicine_name": "타이레놀",
  "storage_stock": 280,
  "dispensing_stock": 23,
  "released_event_ids": [2]
}

released_event_ids 의미:
- 리필 덕분에 REFILL_REQUIRED에서 WAITING으로 바뀐 Event id 목록.
- 예: released_event_ids가 [2]이면 2번 처방전이 이제 제조 가능 상태가 된 것.


화면별 사용 API:

처방전 입력 화면:
- GET /api/medicine/
- POST /api/prescriptions/

제조 현황 화면:
- GET /api/events/

재고 현황 화면:
- GET /api/medicine/

수동 상태 변경 기능:
- POST /api/tasks/status/

수동 리필 완료 기능:
- POST /api/tasks/refill/

ROS2 bridge_node:
- GET /api/tasks/next/
- POST /api/tasks/status/
- POST /api/tasks/refill/


ROS2 Topic 설계:

1. bridge_node → task_manager_node

Topic:
- /robot_task

Type:
- std_msgs/msg/String

내용:
- GET /api/tasks/next/ 응답 JSON 전체를 문자열로 publish

예시:
{
  "event_id": 1,
  "prescription_name": "김철수",
  "status": "WAITING",
  "items": [
    {
      "medicine_name": "타이레놀",
      "quantity": 2,
      "order": 1,
      "item_status": "READY",
      "refill_needed": false,
      "storage_location": "123,456",
      "dispensing_location": "234,567",
      "lid_type": "hole",
      "storage_stock": 300,
      "dispensing_stock": 98
    }
  ]
}


2. task_manager_node → bridge_node

Topic:
- /task_status_request

Type:
- std_msgs/msg/String

내용:
- Event 전체 상태 변경 요청

예시:
{
  "event_id": 1,
  "status": "PROCESSING"
}

예시:
{
  "event_id": 1,
  "status": "DONE"
}


3. task_manager_node 또는 robot_node → bridge_node

Topic:
- /refill_complete

Type:
- std_msgs/msg/String

내용:
- 리필 완료 후 재고 갱신 요청

예시:
{
  "medicine_name": "타이레놀",
  "amount": 20
}


핵심 정리:

Event.status:
- 처방전 전체 상태

EventItem.status:
- 처방전 안의 약 하나하나의 리필 필요 여부

리필 필요 판단 기준:
- EventItem.quantity > Medicine.dispensing_stock
  → EventItem.status = REFILL_REQUIRED

- EventItem.quantity <= Medicine.dispensing_stock
  → EventItem.status = READY

min_stock:
- 삭제됨.
- 더 이상 사용하지 않음.
