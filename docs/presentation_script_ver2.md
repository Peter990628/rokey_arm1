## 4. 주제 선정 배경과 기획 의도

약국 조제 업무는 정확성이 매우 중요하지만, 반복 작업으로 인한 피로와 작업 속도의 한계가 있고, 작은 실수도 오조제로 이어질 수 있습니다.

저희는 약통 운반, 의약품 리필, 스크래퍼와 종이봉투 처리 같은 반복적인 물리 공정을 로봇이 보조하도록 구성했습니다.

다만 의약품의 종류와 수량에 대한 최종 판단은 사람이 맡도록 했습니다. 즉, 약사를 대체하는 시스템이 아니라 약사가 최종 검수와 복약 상담 같은 전문 업무에 더 집중하도록 돕는 것이 핵심 목적입니다. 

자동화 장치로 인해 조제투약 오류를 줄이고 환자와의 접촉 시간을 늘릴 수 있는 등 약사 본연의 역할에 충실하게 다가설 수 있다는 점에서 돈을 써서라도 도입 할 수 밖에 없음

전 세계 약국 자동화 장치 시장 규모는 2025년 34억 7천만 달러로 평가되었으며, 2026년 38억 4천만 달러에서 2034년까지 90억 달러로 성장하여 예측 기간 동안 CAGR 11.21%를 나타낼 것으로 예상됩니다.

## 5. 프로젝트 구현 내용

전체 시스템은 브라우저 기반 UI, Flask, Django Backend와 PostgreSQL DB, ROS 2 통신, 그리고 로봇 제어 노드로 구성됩니다.

처방이 등록되면 Backend가 조제기 재고를 판단하고, 필요한 경우 리필 공정을 먼저 수행합니다. 이후에는 스크래퍼 준비, 약사의 검수, 종이봉투 포장과 수납 작업으로 이어집니다.

팀에서는 웹과 DB, 리필, 스크래퍼와 포장, 봉투 수납을 나누어 개발한 뒤 마지막에 통합했습니다.

## 6. ROS 2 노드 그래프

이 화면은 실행 중인 ROS 2 노드와 토픽의 연결 관계를 확인한 그래프입니다.

[가리키기] 왼쪽의 `task_manager_bridge`가 Backend 정보를 ROS 2 토픽으로 전달하고, 중앙의 `robot_total`이 작업 조건과 실행 순서를 관리합니다.

오른쪽의 `dsr_controller2`는 실제 로봇 명령을 받는 서비스 서버입니다. 주변의 많은 토픽은 관절 상태, 속도, 토크와 같은 두산 로봇 상태 정보입니다.

그래프는 복잡해 보이지만 핵심은 Backend 정보 수신, 통합 노드의 작업 판단, Controller를 통한 로봇 실행의 세 단계입니다.

## 7. 주요 ROS 2 토픽

토픽은 크게 작업 데이터와 설비 간 상태 신호로 나눌 수 있습니다.

`events`, `medicine`, `refill_required_medicine`은 `task_manager_bridge`가 Backend에서 조회한 처방, 약품, 리필 대상 정보를 JSON 문자열로 발행합니다.

`robot_enable`은 작업 시작 허가, `scraper_ready`는 스크래퍼가 배출구에 도착했다는 신호, `dispensing_done`은 조제기의 약 배출 완료 신호입니다.

마지막으로 `robot_status`는 통합 노드의 현재 상태를, `task_done`은 종이봉투 수납까지 완료되었음을 알립니다.

세부 행을 모두 읽기보다는 JSON 토픽은 작업 데이터를, Bool 토픽은 단계별 완료와 허가 신호를 전달한다고 이해하시면 됩니다.

## 9. 주요 JSON 형식

왼쪽은 처방 이벤트 데이터입니다. 이벤트 ID, 처방 이름, 전체 상태와 함께 각 약의 이름, 수량, 순서, 준비 상태가 들어 있습니다.

오른쪽은 리필 대상 약품 데이터입니다. 약 이름과 적재소 좌표, 조제기 좌표, 서랍 좌표, 그리고 약병 끝점을 계산하기 위한 오프셋을 포함합니다.

약품별 정보를 하나의 JSON 객체로 묶었기 때문에 여러 약이 부족해도 각 약의 좌표와 오프셋이 서로 섞이지 않습니다.

## 10. 핵심 기술 I - 정밀 위치 제어

반복 작업을 위해 서랍 손잡이, 적재소의 약통 위치, 조제기 투입 위치와 폐기 위치를 미리 티칭했습니다.

넓은 구간을 이동할 때는 관절 이동인 `movej`를 사용하고, 물체를 넣거나 빼는 구간에는 직선 이동인 `movel`을 사용했습니다.

또한 약통을 고정대에 삽입하거나 오프너를 뚜껑에 접촉시키는 과정에는 순응 제어와 힘 제어를 적용해 작은 위치 오차를 흡수했습니다.

## 11. 위치 제어 적용 장면

이 장면은 티칭 좌표를 기준으로 로봇이 약통과 작업 지그에 접근하는 과정입니다.

작업물에 곧바로 접근하지 않고 안전한 중간 자세를 거친 뒤, 마지막 접촉 구간만 낮은 속도의 직선 이동으로 수행하도록 했습니다.

## 12. 뚜껑 위치 탐색

고정 좌표만으로 뚜껑 위치를 찾으면 약통 배치 오차 때문에 오프너가 빗나갈 수 있습니다.

그래서 로봇을 X축 방향으로 조금씩 이동시키면서 힘 센서 값의 변화를 확인하고, 접촉이 감지되면 뚜껑 위치를 찾은 것으로 판단했습니다.

즉, 좌표로 근처까지 이동한 뒤 힘 센서로 실제 접촉점을 보정하는 방식입니다.

## 13. 약통 고정대 삽입

뚜껑을 열 때 약통 본체가 함께 회전하지 않도록 약통을 고정대 안으로 눌러 넣습니다.

이 과정에서는 아래 방향으로 목표 힘을 주는 순응 제어를 사용하고, 현재 Z 좌표가 목표 범위에 들어왔는지도 함께 확인합니다.

접촉 힘과 위치 조건을 함께 사용해 약통이 충분히 고정되기 전에 다음 동작으로 넘어가는 것을 막았습니다.

## 14. 돌려서 여는 뚜껑

SPIN 방식 뚜껑은 그리퍼의 한 번 회전만으로 완전히 열리지 않을 수 있습니다.

그래서 일정 각도만큼 회전한 뒤 그리퍼를 풀고 원위치로 돌아와 다시 잡는 동작을 반복하도록 구성했습니다.

이 방식으로 로봇 관절의 회전 한계를 피하면서도 뚜껑을 여러 차례 연속으로 돌릴 수 있습니다.

## 15. 개봉 확인과 재시도

회전 후에는 힘과 이동 결과를 확인해 뚜껑이 실제로 열렸는지를 판단합니다.

열리지 않았다고 판단되면 그리퍼를 다시 잡고 개봉 동작을 재시도하며, 정해진 최대 횟수를 넘으면 작업을 중단합니다.

성공 여부를 확인하지 않고 다음 공정으로 진행하는 상황과 무한 반복을 함께 방지한 부분입니다.

## 16. 핵심 기술 II - 정밀 동작과 신뢰성

두 번째 핵심 기술은 약병 기울임 제어와 파지 확인입니다.

약을 부을 때는 약병의 중심이 아니라 약병 입구를 가상 회전축으로 설정해, 기울이는 동안에도 입구가 조제기 투입구에서 크게 벗어나지 않도록 했습니다.

또한 디지털 입력으로 약통 파지를 확인하고, 파지에 실패하면 그리퍼를 열고 다시 접근하도록 구성했습니다.

## 17. 가상 회전축 계산

[가리키기] 왼쪽 사진의 병 입구가 저희가 고정하려는 가상 회전축입니다.

현재 TCP의 위치와 자세, 약병 끝점 오프셋을 이용해 병 입구의 Base 좌표를 계산합니다. 그다음 회전행렬을 적용해 목표 자세와 TCP 위치를 함께 다시 계산합니다.

단순히 손목 각도만 바꾸는 것이 아니라 위치와 자세를 동시에 보정하기 때문에, 병을 단계적으로 기울여도 입구가 투입구 근처에 유지됩니다.

## 18. 핵심 기술 III - 통신과 데이터 연동

세 번째 핵심 기술은 ROS 2와 데이터베이스의 연동입니다.

`task_manager_bridge`는 처방과 약품 정보를 조회해 토픽으로 전달하고, `robot_total`은 수신한 상태와 좌표를 기준으로 작업 큐를 구성합니다.

여기서 Bridge는 GET과 발행만 담당합니다. 실제 리필이 끝난 뒤의 재고 갱신 요청은 통합 로봇 노드가 REST API로 Backend에 전달합니다.

## 19. 데이터베이스 상태 관리

상단 표에는 각 약의 재고, 뚜껑 종류, 적재소와 조제기 좌표 등이 미리 저장되어 있습니다.

처방전인 Event가 등록되면 필요한 약들이 EventItem으로 생성되고, 재고 판단 결과에 따라 각 항목의 상태가 `READY` 또는 `REFILL_REQUIRED`로 기록됩니다.

이 상태를 기준으로 로봇은 리필이 필요한 약만 골라서 처리하고, 모든 약이 준비되어 있으면 공통 조제·포장 단계로 넘어갑니다.

## 20. 웹 대시보드

약사는 웹 대시보드에서 현재 제조 중인 처방과 완료된 처방을 한눈에 확인할 수 있습니다.

Backend의 작업 상태가 변경되면 카드의 위치와 화면 내용도 함께 갱신됩니다.

뒤의 수행 결과 부분에서 재고, 제조 현황, 처방 입력 화면까지 다시 자세히 보여드리겠습니다.

## 21. 활용 장비와 개발 환경

하드웨어는 두산 M0609 협동로봇, 디지털 출력 기반 그리퍼, 조제기 서랍장, 약 적재소와 스크래퍼를 사용했습니다.

약통 고정대와 거치대, 뚜껑 오프너, 툴 거치대는 공정에 맞게 직접 설계하고 제작했습니다.

소프트웨어는 Ubuntu 22.04, ROS 2 Humble과 Python을 사용했고, 로봇 제어에는 `DSR_ROBOT2`, 웹과 DB에는 Flask와 Django를 사용했습니다.

## 22. 자체 제작 지그 설계

화면은 직접 모델링한 스크래퍼 거치대, 약통 고정대, 약통 거치대와 툴 거치대입니다.

소프트웨어 좌표 보정만으로 해결하기 어려운 흔들림과 회전을 지그가 기구적으로 제한하도록 설계했습니다.

특히 로봇이 반복 접근할 수 있도록 각 도구의 위치와 방향을 일정하게 유지하는 데 초점을 맞췄습니다.

## 23. 실제 제작 장비

왼쪽부터 실제 M0609와 그리퍼, 스크래퍼, 약통 오프너, 그리고 툴 거치대입니다.

각 장비는 기능별 유닛 테스트를 먼저 진행한 뒤 전체 작업 공간에 배치했습니다.

## 24. 조제기와 약통 지그

왼쪽은 약을 투입할 조제기 서랍장이고, 오른쪽은 약통 거치대와 고정대입니다.

거치대는 약통의 대기 위치를 일정하게 만들고, 고정대는 뚜껑을 열 때 약통 본체가 같이 회전하는 것을 막습니다.

## 25. 전체 작업 환경

이 사진은 로봇, 적재소, 오프너, 고정대와 조제기를 하나의 작업 공간에 배치한 모습입니다.

실제 설치 후에는 로봇의 도달 범위뿐 아니라 팔꿈치와 손목이 주변 장치에 간섭하는지도 확인하면서 배치를 여러 차례 수정했습니다.

## 26. 작업 공간 배치 설계

초기에는 이 배치도를 기준으로 로봇 팔, 적재소 선반, 조제 서랍, 툴 거치대와 봉투 거치대의 위치를 계획했습니다.

이후 실제 로봇 자세와 도달 범위를 확인하면서 장치 간 거리와 접근 방향을 조정하고 좌표를 다시 티칭했습니다.

## 28. 전체 Flowchart

처방전이 접수되면 먼저 DB가 조제기 안의 재고로 처방 수량을 처리할 수 있는지 판단합니다.

## 29. 시스템 아키텍처

상단은 처방 UI, Django REST Backend와 PostgreSQL DB로 구성된 업무·데이터 계층입니다.

조제기와는 `scraper_ready`와 `dispensing_done` 신호를 교환합니다. 리필이 완료되면 통합 노드가 Backend에 완료 정보를 POST합니다.

하단에서는 `DSR_ROBOT2`가 서비스 클라이언트, `dsr_controller2`가 서비스 서버 역할을 하며 실제 M0609와 그리퍼를 제어합니다.

최종 판단을 사람에게 남겨 둔 이유는 의약품 작업의 안전성과 신뢰성을 확보하기 위해서입니다.

검수가 끝난 약이 봉투에 들어가면 스크래퍼를 원위치로 반환하고, 종이봉투를 거치대에서 분리해 수납대에 올려놓습니다.

## 43. 웹 페이지 오류 처리

웹에서는 잘못된 입력이 처방 데이터로 등록되지 않도록 여러 검증을 적용했습니다.

같은 약의 중복 선택을 막고, 수량은 1개에서 50개 사이로 제한하며, 환자 이름이나 약 정보가 없으면 서버 요청 자체를 보내지 않습니다.

서버 통신이 실패하면 오류를 안내하되 화면 전체가 멈추지 않도록 `try-catch`로 처리하고, 검수 완료는 확인 팝업을 한 번 더 거치도록 했습니다.

표의 모든 항목을 읽기보다는 사용자 입력, 서버 통신, 실수 방지의 세 단계로 예외 처리를 구성했다는 점이 핵심입니다.

## 44. 로봇 제어 노드 예외 처리

로봇 쪽에서는 재고가 0 이하이거나 필수 데이터가 비정상이면 작업을 시작하지 않습니다.

약통 파지는 센서 신호를 확인하고 실패하면 제한된 횟수만 재시도합니다. 고정, 뚜껑 탐색과 힘 제어에는 timeout을 적용하고, 기준을 넘는 힘이나 토크가 감지되면 즉시 동작을 멈춥니다.

실패한 상태에서 다음 공정을 계속 수행하면 더 큰 충돌이나 중복 투입으로 이어질 수 있기 때문에, 성공이 확인된 단계만 다음 단계로 진행하도록 구성했습니다.


## 47. 기술적 문제 해결

첫 번째 문제는 작업 중 TCP를 바꿨을 때 좌표가 예상대로 변환되지 않은 것이었습니다. 이를 도구별 좌표 오프셋과 약병 끝점 기반 계산으로 해결했습니다.

두 번째로 약통이 고정대 안에서 계속 회전하는 문제는 힘 제어 회전 대신 제한된 `movel` 동작과 전용 고정 구조를 적용했습니다.

좁은 작업 공간에서 발생한 충돌은 수평 이동 전에 목표 지점보다 높은 안전 높이로 먼저 올라가는 경로를 추가해 줄였습니다.

마지막으로 PULL 방식 뚜껑은 초기 지렛대 방식이 불안정해 전용 약통 뚜껑 오프너를 새로 설계했습니다.

## 48. 약통 파지와 고정 구조 개선

처음에는 왼쪽처럼 하나의 집게 형태로 약통 파지와 고정을 함께 해결하려 했습니다.

하지만 TCP 변환과 반복 정밀도에 문제가 있어, 오른쪽과 같이 약통을 잡는 기능과 고정하는 기능을 분리하고 각 도구에 맞는 좌표 오프셋을 적용했습니다.

기능을 분리하면서 구조가 단순해지고 반복 동작도 더 안정적으로 만들 수 있었습니다.

## 49. PULL 오프너 개선

초기에는 송곳으로 뚜껑을 뚫고 지렛대 원리로 여는 방식을 시도했지만, 접촉 위치가 일정하지 않고 뚜껑이 손상될 가능성이 있었습니다.

그래서 오른쪽처럼 뚜껑 테두리에 직접 걸어 위로 당기는 전용 오프너를 제작했습니다.

소프트웨어 좌표만 계속 수정하기보다 문제의 원인을 기구적으로 제한하는 방향으로 해결한 사례입니다.

## 50. 프로젝트 성과

좌표 티칭을 기반으로 처방 데이터 수신부터 리필, 조제 대기, 검수, 포장까지 이어지는 End-to-End 공정을 구현했습니다.

약통 뚜껑을 두 방식으로 여는 과정과 조제기 서랍을 열고 약을 투입하는 동작, 스크래퍼와 종이봉투 작업을 실제 로봇으로 검증했습니다.

또한 상태 확인, timeout과 제한된 재시도를 적용해 실패 시 무한 반복하지 않는 기본적인 중단과 복구 구조를 확보했습니다.

## 51. 완성도 평가와 개선점

기획 대비 완성도는 10점 만점에 9점으로 평가했습니다.

핵심 공정을 실제 로봇으로 구현했지만, 실패 지점부터 자동으로 복구하는 로직과 실패 이력 로깅·알림은 더 보완해야 합니다.

또한 적재소의 약통을 사람이 정렬해야 하는 작업을 줄이고, 다양한 약통 규격에 대응할 수 있는 가변형 고정 장치를 설계하는 것이 다음 개선 과제입니다.

하드웨어도 구조 해석을 바탕으로 부품 수와 동작 저항을 줄일 필요가 있습니다.

## 52. 회고 I

실제 로봇은 좌표값만 입력한다고 원하는 동작이 바로 나오지 않았고, 기구학적 제약과 TCP의 의미를 함께 이해해야 했습니다.

특히 작업 중 TCP 설정을 자유롭게 바꾸기 어렵다는 점을 늦게 파악해 초기 디버깅에 많은 시간을 사용했습니다.

하지만 이 문제를 해결하면서 두산 로봇 라이브러리를 직접 분석하고, ZYZ 오일러각과 회전행렬이 실제 자세에 어떻게 반영되는지를 구체적으로 이해할 수 있었습니다.

## 53. 회고 II

각도 하나가 잘못되면 로봇이 예상과 반대 방향으로 크게 움직일 수 있었고, 비전 없이 고정 좌표에 의존하다 보니 1밀리미터의 오차도 작업 성공 여부에 영향을 줬습니다.

하드웨어와 소프트웨어를 함께 연결하는 과정에서 많은 문제가 있었지만, 로그와 실제 동작을 비교하며 원인을 찾는 실전 디버깅 역량을 키울 수 있었습니다.

향후에는 비전 기반 위치 보정과 추가 센서를 적용해 변화하는 작업 환경에도 대응하는 시스템으로 발전시키고 싶습니다.

# 예상 질문과 답변

## 왜 비전을 사용하지 않았나요?

이번 프로젝트는 9일이라는 제한된 기간 안에 고정된 약국 공정을 End-to-End로 구현하는 것이 목표였기 때문에 좌표 티칭과 전용 지그를 우선 적용했습니다. 실제 환경으로 확장하려면 약통과 봉투의 위치 오차를 보정할 비전 또는 추가 센서가 필요합니다.

## `robot_enable`이 비상 정지 역할을 하나요?

아닙니다. `robot_enable`은 신규 작업의 시작을 허가하거나 차단하는 소프트웨어 운영 신호입니다. 사람 보호를 위한 즉시 정지는 Doosan Controller의 E-Stop, Protective Stop과 Safety I/O가 담당해야 합니다.

## 왜 ROS 메시지를 JSON String으로 보냈나요?

Backend가 JSON을 사용하고 개발 기간이 짧아 REST 응답을 빠르게 연결하기 위해 `std_msgs/String`을 사용했습니다. 향후에는 필드 타입과 필수 값을 ROS 계층에서도 검증할 수 있도록 전용 메시지 인터페이스로 변경하는 것이 좋습니다.

## `task_manager_bridge`도 DB를 수정하나요?

아닙니다. `task_manager_bridge`는 `/api/events/`와 `/api/medicine/`을 GET으로 읽고 ROS 2 토픽을 발행하는 읽기 전용 브릿지입니다. 리필 완료에 따른 Backend 갱신은 `robot_total`이 별도의 REST POST로 수행합니다.

## 모든 약이 `READY`이면 스크래퍼 신호를 Bridge가 바로 보내나요?

Bridge가 스크래퍼 신호를 직접 보내지는 않습니다. Bridge는 Event 정보를 발행하고, Event 전체 상태가 `WAITING`이면 `robot_total`이 포장 작업을 시작합니다. 실제로 스크래퍼를 집어 배출구에 도착한 뒤 `robot_total`이 `scraper_ready=true`를 발행합니다.

## 로봇 작업이 실패하면 자동으로 다시 실행되나요?

약통 파지처럼 물리 상태를 다시 확인할 수 있는 일부 단계는 제한된 횟수만 재시도합니다. 기준 횟수를 넘기거나 힘과 토크가 비정상이면 작업을 중단합니다. 물체 상태를 확인하지 않고 무조건 재시도하면 충돌이나 중복 투입이 발생할 수 있기 때문입니다.

## 약병을 기울일 때 입구를 유지한 방법은 무엇인가요?

현재 TCP의 회전행렬과 약병 끝점 오프셋을 이용해 병 입구의 Base 좌표를 계산했습니다. 이 점을 가상 회전축으로 유지하도록 회전 후 TCP 위치와 자세를 함께 계산해 `movel` 목표를 생성했습니다.

## SPIN과 PULL 방식은 어떻게 선택하나요?

Backend의 약품 정보에 저장된 `lid_type`을 기준으로 분기합니다. SPIN이면 그리퍼로 뚜껑을 반복 회전하고, PULL이면 전용 오프너를 장착해 뚜껑에 걸고 당기는 공정을 실행합니다.

## 현재 가장 먼저 개선해야 할 부분은 무엇인가요?

실패 단계와 원인을 영속적으로 저장하는 로깅·알림 시스템, 실패 지점부터 안전하게 재개하는 복구 로직이 우선입니다. 이후에는 비전 기반 위치 보정, 추가 파지 센서와 다양한 약통 규격에 대응하는 가변형 지그가 필요합니다.
