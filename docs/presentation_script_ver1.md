## 4. 주제 선정 배경 및 기획 의도

약국 조제 업무는 정확성이 매우 중요하지만, 반복 작업으로 인한 피로와 처리 속도의 한계가 있습니다.

저희는 사람이 모든 물리 작업을 반복하는 대신, 약통 이동과 리필, 포장 같은 반복 공정을 로봇이 보조하도록 구성했습니다.

다만 의약품은 안전이 중요한 만큼 최종 검수는 사람이 담당하도록 역할을 분리했습니다. 즉, 사람을 완전히 대체하는 것이 아니라 약사가 전문적인 판단과 검수에 더 집중하도록 돕는 것이 핵심 목적입니다.

## 5. 프로젝트 구현 내용

전체 구조는 브라우저, Flask UI, Django Backend와 DB, ROS 2, 그리고 로봇 제어 노드로 구성됩니다.

처방이 등록되면 Backend가 재고를 확인하고, 로봇이 필요한 리필과 포장 작업을 순서대로 실행합니다.

웹과 DB는 처방 및 재고를 담당하고, 로봇 쪽은 약통 개봉, 조제기 리필, 스크래퍼와 종이봉투 작업으로 역할을 나누어 개발했습니다.

## 6. ROS 2 노드 그래프

이 화면은 실제 ROS 2 노드와 토픽의 연결 관계를 확인한 그래프입니다.

[가리키기] 중앙의 `robot_total`이 통합 제어 노드이고, 왼쪽의 `task_manager_bridge`에서 작업 정보를 받습니다. 오른쪽의 `dsr_controller2`를 통해 실제 로봇 서비스로 명령이 전달됩니다.

화면이 복잡해 보이지만 핵심 흐름은 Backend 정보 수신, 작업 판단, 로봇 명령 실행의 세 단계입니다.

## 7. 주요 ROS 2 토픽

토픽은 크게 세 종류로 나눌 수 있습니다.

첫째, `events`, `medicine`, `refill_required_medicine`은 Backend에서 받은 처방과 약품 정보를 JSON으로 전달합니다.

둘째, `robot_enable`, `scraper_ready`, `dispensing_done`은 로봇과 조제기 사이의 작업 허가 및 완료 신호입니다.

셋째, `robot_status`와 `task_done`은 현재 로봇 상태와 최종 작업 완료를 외부에 알립니다.

세부 항목을 모두 읽기보다는, JSON 토픽은 작업 데이터를 전달하고 Bool 토픽은 설비 간 신호를 교환한다고 이해하시면 됩니다.

## 8. 네트워크 구성도

운영자 PC에서는 Flask 기반 UI가 실행되고, Backend PC의 Django REST API와 HTTP로 통신합니다.

Robot PC의 `task_manager_bridge`는 Backend를 조회해 ROS 2 토픽을 발행하고, `robot_total`이 이 정보를 받아 작업을 수행합니다.

로봇 명령은 `DSR_ROBOT2`의 서비스 클라이언트를 거쳐 `dsr_controller2`와 Doosan Controller로 전달됩니다.

ROS 2 장치끼리는 같은 Domain ID와 DDS Discovery 설정을 사용해야 다른 PC에서도 토픽을 확인할 수 있습니다.

## 9. 주요 JSON 형식

왼쪽은 처방 이벤트 데이터입니다. 처방 ID, 환자 이름, 전체 상태와 필요한 약의 이름, 수량, 순서를 포함합니다.

오른쪽은 리필이 필요한 약의 데이터로, 적재소와 조제기의 좌표, 서랍 좌표, 그리고 약병 끝점 offset을 포함합니다.

한 약품의 좌표와 offset을 하나의 객체로 묶어 보내기 때문에 약이 여러 개여도 서로 좌표가 섞이지 않도록 구성했습니다.

## 10. 핵심 기술 I - 정밀 위치 및 힘 제어

반복 작업을 위해 서랍 손잡이, 적재소, 조제기와 폐기 위치를 미리 티칭했습니다.

큰 구간 이동에는 `movej`를 사용하고, 물체를 넣거나 빼는 직선 구간에는 `movel`을 사용했습니다.

약통을 고정대에 삽입하거나 뚜껑 오프너를 접촉시키는 과정에서는 위치 오차를 흡수하기 위해 순응 제어와 힘 제어를 적용했습니다.

## 11. 힘 제어 적용 장면

사진은 실제 작업 환경에서 약통과 지그의 위치를 맞추며 힘 제어를 시험한 장면입니다.

고정 좌표만 사용하면 작은 제작 오차에도 충돌하거나 삽입이 멈출 수 있어, 접촉 구간만 힘 제어를 사용하고 나머지는 위치 제어로 동작하도록 구분했습니다.

## 12. 자체 제작 지그

약통 형태와 뚜껑 종류에 맞추기 위해 약통 고정대, 뚜껑 오프너, 툴 거치대 등을 직접 설계하고 제작했습니다.

로봇 프로그램만으로 해결하기 어려운 흔들림과 회전 문제를 지그로 제한해 동작의 반복성을 높였습니다.

## 13. 핵심 기술 II - 기울임과 파지 신뢰성

약을 부을 때는 약병의 중심이 아니라 병 입구를 가상 회전축으로 설정했습니다.

현재 TCP 자세와 병 끝점 offset을 회전행렬로 계산해, 병을 기울여도 입구가 조제기 투입구에서 크게 벗어나지 않도록 했습니다.

또한 약통 파지는 디지털 입력으로 확인하며, 신호가 들어오지 않으면 그리퍼를 열고 다시 접근합니다. 통합 코드에서는 재시도 횟수를 제한해 무한 반복도 방지했습니다.

## 14. 핵심 기술 III - 데이터 연동과 상태 관리

Backend의 처방과 약품 정보는 ROS 2 토픽으로 전달되고, 로봇은 수신한 좌표와 상태를 기준으로 작업 큐를 구성합니다.

리필이 완료되면 로봇 제어 노드가 REST API를 호출해 적재소와 조제기 재고를 갱신합니다.

즉, 화면상의 재고 정보와 실제 로봇 동작이 하나의 작업 흐름으로 연결되도록 구성했습니다.

## 15. 웹 대시보드 미리보기

약사는 웹 대시보드에서 제조 중인 처방과 완료된 처방, 재고 상태를 확인할 수 있습니다.

뒤의 수행 결과 부분에서 각 화면을 조금 더 자세히 보여드리겠습니다.

## 16. 활용 장비 및 개발 환경

하드웨어는 두산 M0609, 디지털 출력 기반 그리퍼, 조제기, 약 적재소, 스크래퍼와 자체 제작 지그를 사용했습니다.

소프트웨어는 Ubuntu 22.04, ROS 2 Humble, Python을 사용했고, 로봇 제어에는 `DSR_ROBOT2`, UI와 Backend에는 Flask와 Django를 사용했습니다.

## 17. 주요 장비

왼쪽부터 M0609와 그리퍼, 스크래퍼, 약통 오프너, 툴 거치대입니다.

각 도구는 로봇이 반복적으로 접근할 수 있도록 위치를 고정하고, 도구별 TCP와 좌표를 적용했습니다.

## 18. 조제기 및 약통 지그

왼쪽은 약을 투입할 조제기이고, 오른쪽은 약통 거치대와 고정대입니다.

고정대는 뚜껑을 열 때 약통 본체가 함께 회전하지 않도록 잡아주는 역할을 합니다.

## 19. 전체 작업 환경

## 22. Flowchart

처방전이 접수되면 먼저 DB가 조제기 재고를 판단합니다.

[가리키기] 재고가 부족한 경우에는 서랍을 열고, 적재소에서 약통을 가져와 뚜껑을 연 뒤 조제기에 리필합니다. 재고가 충분하면 이 조건부 경로는 생략됩니다.

이후 공통 과정으로 스크래퍼를 조제기 배출구에 준비하고, 약사의 검수 완료 신호를 받은 뒤 약을 봉투에 붓고 종이봉투를 수납대로 옮깁니다.

보라색은 필요할 때만 실행하는 리필 경로이고, 분홍색은 모든 처방에서 수행하는 포장 경로입니다.

## 23. System Architecture

상단은 처방 UI와 Django Backend, PostgreSQL DB로 구성된 업무 및 데이터 계층입니다.

중간에서 `task_manager_bridge`가 Backend 데이터를 GET으로 조회해 ROS 2 토픽으로 변환하고, `robot_total`이 개봉, 리필, 포장 작업을 판단하고 순서대로 실행합니다.

조제기와는 `scraper_ready`, `dispensing_done` 신호를 교환합니다.

하단에서는 `DSR_ROBOT2`가 서비스 클라이언트, `dsr_controller2`가 서비스 서버 역할을 하며 실제 M0609와 그리퍼를 제어합니다.

## 28. 구현된 네트워크 구조

앞에서 설명한 네트워크 구조를 실제 구현 기준으로 다시 정리한 화면입니다.

Backend와 Bridge는 HTTP로 연결되고, Bridge와 통합 노드는 ROS 2 토픽으로 연결됩니다. 로봇 명령은 ROS 2 서비스를 통해 Controller로 전달됩니다.

업무 데이터, 작업 신호, 로봇 명령을 서로 다른 통신 역할로 분리한 것이 특징입니다.

## 29. 대시보드

메인 화면은 제조 중인 처방과 제조 완료된 처방을 두 영역으로 나누어 보여줍니다.

환자 이름, 처방 번호와 약품을 카드 형태로 확인할 수 있으며, Backend의 작업 상태를 기준으로 화면이 갱신됩니다.

왼쪽 사이드바를 통해 대시보드, 재고 현황, 제조 현황, 처방전 입력 화면으로 이동할 수 있습니다.

## 30. 재고 현황

재고 화면에서는 약품 이름, 적재소 기준 현재 수량, 임계값과 상태를 확인할 수 있습니다.

현재 수량이 기준 이하로 내려가면 부족 상태를 표시해 작업자가 사전에 확인할 수 있도록 했습니다.

## 31. 제조 현황 및 검수

제조 현황에서는 처방별 등록 시간, 약품과 수량, 진행 상태를 확인합니다.

제조 중인 처방에는 검수 완료 버튼이 표시됩니다. 약사가 실제 약을 확인한 뒤 버튼을 누르면 Backend의 Event 상태가 완료로 변경됩니다.

최종 판단을 사람에게 남겨 둔 이유는 의약품 작업의 신뢰성을 확보하기 위해서입니다.

## 32. 처방전 입력

물리적으로는 로봇과 작업자의 충돌, 지그 사이의 끼임, TCP 설정 오류와 물체 낙하를 주요 위험으로 보았습니다.

코드에서는 작업 전 Tool과 TCP를 설정하고 Ready 자세에서 시작하며, 파지 입력과 최종 봉투 좌표를 확인합니다.

다만 사람 보호는 Python 예외 처리만으로 보장할 수 없습니다. 실제 작업에서는 Controller의 비상 정지, 충돌 감지와 안전 설정을 함께 사용해야 합니다.

의약품 측면에서는 약 이름과 재고 상태를 DB에서 확인하고, 최종 검수는 사람이 수행하도록 구성했습니다.

## 39. 시스템 및 운영 안전

잘못된 JSON이 들어오면 `try-except`로 해당 메시지를 무시하고 로봇 작업에 사용하지 않습니다.

또한 기본 `auto_start`를 false로 설정해 명시적인 enable 신호가 있어야 작업을 시작합니다.

여기서 `robot_enable`은 작업 허가 신호이지 안전 정격 정지 기능은 아닙니다. 비상 상황의 즉시 정지는 반드시 Controller와 E-stop 같은 하드웨어 안전 기능이 담당해야 합니다.

## 40. 문제 해결 과정

개발 중 가장 먼저 겪은 문제는 TCP 전환과 실제 좌표가 예상대로 맞지 않는 것이었습니다. 이를 도구별 offset과 병 끝점 기반 좌표 계산으로 해결했습니다.

약통이 고정대에서 함께 회전하는 문제는 힘 제어 회전 대신 제한된 `movel` 회전과 전용 지그를 사용했습니다.

조제기로 이동할 때 충돌한 문제는 수평 이동 전에 조제기보다 높은 안전 높이로 먼저 이동하도록 경로를 변경했습니다.

PULL 방식 뚜껑은 초기 지렛대 방식이 불안정해 전용 오프너를 새로 설계했습니다.

## 41. 오프너 개선 결과

왼쪽은 초기 형태이고 오른쪽은 개선한 오프너입니다.

기존 방식의 흔들림과 접촉 위치 문제를 줄이기 위해 약통과 오프너의 위치를 기구적으로 제한하도록 구조를 변경했습니다.

소프트웨어 조정만 반복하기보다 하드웨어 구조를 함께 개선한 것이 중요한 해결 과정이었습니다.

## 42. 프로젝트 성과

좌표 티칭을 기반으로 처방 데이터 수신부터 리필, 조제 대기, 포장까지 이어지는 End-to-End 공정을 구현했습니다.

특히 약통 뚜껑을 여는 작업, 조제기 서랍을 열고 약을 투입하는 작업, 종이봉투를 수납대로 옮기는 작업을 실제 로봇으로 검증했습니다.

또한 파지 확인, timeout, 최대 재시도와 오류 상태 전환을 적용해 실패 후 무한 반복하지 않는 기본 복구 구조를 확보했습니다.

## 43. 회고

마지막으로 비전 없이 고정 좌표에 의존하면서 1밀리미터 수준의 오차도 작업 성공 여부에 큰 영향을 준다는 것을 경험했습니다.

# 예상 질문과 답변

## 왜 비전을 사용하지 않았나요?

이번 프로젝트는 제한된 기간 안에 고정된 약국 공정을 구현하는 것이 목표였기 때문에 좌표 티칭과 전용 지그를 우선 사용했습니다. 다만 실제 환경에서는 위치 오차와 물체 상태를 확인하기 위해 비전 또는 추가 센서가 필요합니다.

## `robot_enable`이 비상 정지 역할을 하나요?

아닙니다. `robot_enable`은 신규 작업 시작을 허가하거나 차단하는 소프트웨어 신호입니다. 사람 보호를 위한 즉시 정지는 Doosan Controller의 E-stop, Protective Stop과 Safety I/O가 담당해야 합니다.

## 왜 ROS 메시지를 JSON String으로 보냈나요?

Backend가 JSON을 사용하고 개발 기간이 짧아 데이터를 빠르게 연결하기 위해 `std_msgs/String`을 사용했습니다. 향후에는 필드 타입을 엄격하게 검증할 수 있도록 전용 ROS interface로 변경하는 것이 좋습니다.

## 로봇 작업이 실패하면 자동으로 다시 실행되나요?

약통 파지는 제한된 횟수만 재시도합니다. 그 이후의 실패는 `ERROR` 상태로 전환하고 로봇을 disable해 운영자가 물리 상태를 확인하도록 했습니다. 물체 상태가 바뀐 뒤 무조건 자동 재시도하면 충돌이나 중복 투입이 발생할 수 있기 때문입니다.

## 약통을 기울일 때 입구가 움직이지 않게 한 방법은 무엇인가요?

현재 TCP의 회전행렬과 약병 끝점 offset을 이용해 병 입구의 Base 좌표를 계산했습니다. 그 점을 가상 회전축으로 유지하도록 회전 후 TCP 위치와 자세를 함께 계산해 `movel` 목표를 생성했습니다.

## 현재 가장 먼저 개선해야 할 부분은 무엇인가요?

UI와 통합 노드의 상태 메시지 형식을 완전히 통일하고, 작업 상태를 영속화하는 것이 우선입니다. 이후에는 스크래퍼와 종이봉투 파지 센서, 데이터 heartbeat, 비전 기반 위치 보정을 추가하는 것이 필요합니다.
